8. 【发布状态判定】为本次写入飞书多维表格的每条记录判定「发布状态」并回填。
   目的：区分「代码已合并进 main」和「已随正式 release 部署到线上、用户真能用到」——避免宣发了用户还看不到的功能。

   判定工具：/home/node/.openclaw/workspace/tmp/zooclaw-updates/scripts/release_status.py
     - 非 Skill 类：`python3 scripts/release_status.py <owner/repo> <commit_sha>` → 输出 已上线 / 已合并待发版 / 未知
       原理：取该仓库最新【正式 release tag】(排除 draft/prerelease；ecap-workspace 取 `ecap-v*-release`，agent-pack 取 `vX.Y.Z-release` 非 beta)，
       用 GitHub compare API 判断该 tag 是否包含此 commit。包含=已上线，晚于 tag=已合并待发版。
     - ⚠️ Skill 类（信息类型=Skill 上架/更新）：不要用上面的 release tag（ecap-skills 仓库 tag 常年很旧不可靠）。
       改用 ClawHub：`openclaw skills verify <slug>`，看返回 JSON 的 version + createdAt(发布时间)。
       若 ClawHub 最新发布时间 >= 该 commit 日期 且已含本次改动 → 已上线；否则 已合并待发版。
       额外注意：部分 Skill(如 pptx)依赖运行时镜像打包(如 @zooclaw/web-search)，即使 ClawHub 已发布也需运行时重建才生效；存疑标「已合并待发版」并在备注注明原因。
     - 判不出来（无 SHA / API 错 / 拿不准）→ 填「未知/需确认」，不要瞎猜。

   回填：用 bitable batch_update 把每条记录的「发布状态」字段更新为判定结果；
   若为 Skill 或有运行时依赖导致的特殊结论，在「备注」字段补一句原因。
