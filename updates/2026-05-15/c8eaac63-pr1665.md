Fixes unintentional drop of the `?sp=<specialist_id>` query param on post-auth navigation paths. The param signals "auto-hire + prefill this specialist from the landing-page context in localStorage" (ECA-641); when it's dropped, the user lands on a clean `/chat` and `useLandingContextFlow` Phase 1 never fires, so auto-hire + auto-prefill silently break for new users coming from a Specialist landing page through OTP / payment flows.

Four callsites hardcoded `/chat`. They now route through `getLandingChatPath()` which rebuilds `?sp=` from localStorage when the URL has lost it (helper already existed for exactly this case вАФ see `web/src/lib/landing-context.ts:166-179`):

- `web/src/app/[locale]/userguide/UserGuideClient.tsx` вАФ `onLoginSuccess` callback
- `web/src/app/[locale]/pricing/PublicPricingClient.tsx` вАФ `onLoginSuccess` callback
- `web/src/app/landing/LandingClient.tsx` вАФ `handleAuth` (sticky CTA / hero buttons, already-logged-in branch)
- `web/src/app/landing/hooks/useLandingRedirect.ts` вАФ `getRedirectTarget()` default fallback.
