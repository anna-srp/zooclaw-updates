fix(web): surface backend error detail in user-facing error messages (#2843)

## Summary
- Surface backend API error detail in user-facing error toasts/banners
instead of fixed generic text ("Failed to add agent. Please try again.",
"Network error. …"), across ~29 call sites: agents-manager
hire/fire/update/reset, publish/ZooSquare actions, pack purchase flow,
billing (subscription lifecycle, customer portal, invoices, plan renew),
settings data-permissions consent save, Antom payment confirmation poll,
chat interactive cards, onboarding companion select, restart prompt, and
the legacy task-failure bubble.
- New shared helper `withErrorDetail(fallback, err)` in
`web/app/src/lib/query/normalize-error.ts` composes `"<existing
localized/generic prefix>: <backend detail>"`, stripping the prefix's
trailing `.`/`。` to avoid an `".:"` seam; the generic text remains the
fallback when no detail is available (e.g. plain network failures).
`normalizeError` now also preserves plain-string errors so
result-envelope details (`data.error`) flow through the same helper.
- Deliberately unchanged: error boundaries (`ErrorBoundary`,
`app/error.tsx`, `global-error.tsx`) and SSE/WebSocket connection-loss
messaging, where generic wording is intentional and no usable detail
exists.

## Root cause
Both API layers already carry backend detail — `lib/api/backend.ts`
returns `error = data.error || data.detail || data.message || "HTTP
<status>"`, and `lib/api/errors.ts` throws `ApiError` with
message/code/status — but many call sites discarded it via bare `catch
{}`, discarded `(_err)` params, `onError: () => void` callback
signatures (publish hooks), or by reading `result.error` on one branch
but not the sibling branch (`useSubscriptionLifecycle`). Users only ever
saw the hardcoded generic string.

strip, no-detail passthrough) and string-preserving `normalizeError`
