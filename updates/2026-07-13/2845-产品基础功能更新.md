## Summary

- Add requester-authorized Agent Pack Fork discovery for public Official
Packs, direct purchases, entitled hidden industry Packs, and active
Packs in the current organization.
- Create independent Agent Builder projects with immutable source
provenance, confirmed target identity, atomic target ID reservation, and
first-submission binding to a new Pack.
- Gate Fork on a compatible Agent Studio release/workspace and import
the approved source Submission directly from its trusted R2 asset.
- Make Submit recoverable when a Submission is created before its Pack
Test Run association is persisted.
- Replace Open Project/Open Builder entry points with one shared Fork
chooser and confirmation flow, while preserving the default Pack list
API contract.
- Add the implementation design and completed architecture review
findings.

## Rollout notes

- Removing `/agent-builder/projects/open` and the share URL/text import
flow is an intentional breaking cutover. Backend, frontend, and the
Fork-capable Agent Studio release must be coordinated; arbitrary
independent rollout order is not supported for this migration.
- The Fork chooser is controlled by the backend capability endpoint and
remains disabled for a stale or unavailable Agent Studio release.
- The corresponding Fork-capable Agent Studio beta must be available
before end-to-end staging validation.

## Size rationale

The reviewed diff is 3,319 lines after CI exclusions, 319 lines above
the standard limit. This is one cross-layer capability: server-side
entitlement enforcement and source resolution, immutable project/target
identity, Agent Studio compatibility gating, submit recovery, two
frontend entry points, mock support, and regression tests must land
together to avoid an enabled UI with an incomplete authorization or
publication path. The design document is already excluded from the CI
count, and the remaining size is primarily implementation plus coverage
for the security and recovery boundaries.
