## Summary

用户发出的消息要等一会儿才出现在对话框里。根因不是"慢"，而是架构上不可能快：消息列表完全由服务端派生，`useMattermost.sendMessage` 在 `await api.sendPost(...)` **之后**才写 post store，而那个 store 是列表的唯一来源——所以在 REST 往返返回之前，气泡在数据上根本不存在。会话首条更糟：credits 预检先跑，两次串行网络调用之后才画得出东西。

改动：

- **乐观插入 + 对账**。在 `await` 之前插入临时 post，服务端确认后换成真实 post。对账键用 Mattermost 原生的 `pending_post_id`（REST 响应和 WS `posted` 事件都会原样回传，本仓此前 0 引用）。对账逻辑放在 `syncPosts` 内部，REST 响应和 WebSocket echo 谁先到都能幂等收敛，发送路径上不需要处理竞态；另有一条 text/user/channel/附件数 的兜底匹配器，覆盖不回传该字段的服务端。
- **落点在 `useMattermost.sendMessage`**，即主 chat 的发送入口。**范围仅限主 chat**：session thread 和 agent-builder 对话用的是独立的状态容器（`useLiveThread` 的数组 + `locallySentPosts`，按 `root_post_id` 切 thread），不共享 post store，因此拿不到 `syncPosts` 内的 `pending_post_id` 对账，仍是原来的 await-后-回显行为。这两个界面的乐观渲染另行跟进：ECA-1304。本 PR 对它们的唯一改进是收紧了发送结果契约，使失败后草稿恢复在那里也生效。
- **credits 预检不再阻塞首帧**：先画气泡，预检拒绝再撤回。
- **失败处理**：撤回气泡 + 文字回填输入框（此前静默丢失）+ composer banner 说明原因，并保留已上传的附件供重试。同时清掉 waiting-for-reply 状态——否则输入框会被禁用 60 秒，用户拿回了文字却按不
