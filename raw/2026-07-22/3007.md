## 问题

`/chat` 流式输出时对话窗口**从头到尾不自动下滚**，底部「↓」按钮也不出现。

## 先排除的两条

**不是布局问题。** viewport 就是唯一滚动容器，实测 `scrollHeight 87557 / clientHeight 444 / scrollTop 87113`（正好在底部），从它到 `<html>` 每一层祖先（含 `<main class="flex-1 overflow-y-auto">`）都不滚动。

**「↓ 按钮不出现」不构成证据。** 那个按钮只在 `handleScroll` 里重算显隐，而 `handleScroll` 只由 scroll 事件触发；内容在下方增长不产生 scroll 事件，所以它在流式期间**结构上就不可能出现**。这是一个独立 bug，不是主因的线索——我差点把它当决定性证据用。

## 根因：`isAtBottom` 被布局假象锁死

assistant-ui 的流式跟随只有一条路径，在 `useThreadViewportAutoScroll` 里：

```js
else if (autoScroll && store.isAtBottom) scrollToBottom("instant")
```

而 `isAtBottom` **只在 scroll 事件的 handler 里重算**，且把「scrollTop 没往下走、又不在底部」的 scroll 事件一律判成「用户往上滚了」。

我们的流式 markdown 每帧重新解析，写到一半的 token（未闭合的 ``` 围栏、半行列表）先按高的渲染、闭合后塌回去 —— `scrollHeight` 收缩，浏览器**夹紧** `scrollTop`。upstream 把这个夹紧读成用户意图。标志一旦清掉，就再没有任何滚动，handler 再不运行，也就永远翻不回来。

实测（31 条消息，35s 一轮）：

```
t= 4251  scrollTop=87325  scrollHeight=87752  dist=   0   scrollTo=13
t= 5251  scrollTop=87416  scrollHeight=87884  dist=  40   scrollTo=13  ← 最后一次
t=27251  scrollTop=87416  scrollHeight=89287  dist=1443   scrollTo=13
```

MutationObserver 全程触发 **1569** 批变动 —— 回调一直在跑，只是每次都走「不滚」的分支。
